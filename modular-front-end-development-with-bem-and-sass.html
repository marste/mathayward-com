Tutorial: Modular front-end development with BEM and Sass

Wednesday 14th August 2013

Front-end development has come a long way in the past few years with the rise of responsive design and object-oriented CSS. But by far my favourite advancement is in CSS preprocessors, like LESS and Sass.

CSS can get a little tiresome at times. Repeating HEX values, including font-sizes in em’s as well as px’s, over and over and over again. CSS Preprocessors help to alleviate these pains, and some others we may not have thought of.

Not only that, we can save more time by making our code easier to understand using well constructed comments and clear naming conventions on our classes. The BEM methodology gives us this clarity. BEM stands for Block-Element-Modifier and is designed to help modularise front-end development by breaking everything into blocks, then elements and then use modifiers to tweak them.

To apply the BEM methodology, we give each element within a block a class. Consider an unordered list for example:

<ul class="list">
    <li class="list__item">Item 1</li>
    <li class="list__item">Item 2</li>
    <li class="list__item list__item--end">Item 3</li>
</ul>

Our unordered list has the block class “list” while each list item has the element class “list__item”. Only the final list item has the additional modifier class “list__item--end”. Each class follows the BEM syntax:

.BLOCK{__ELEMENT[--MODIFIER]}

Getting started

As with all development, I feel, the easiest (and often quickest) way to learn is to jump straight in.

First things first, we have to change the way we think about building websites. Previously we might build a number of page templates for a site and populate the content within it. But this could lead to bloated stylesheets and highly specific selectors as some templates differed, only slightly, from others.

Designing systems rather than pages means there are no solid page templates; any part of the system can contain any group of components; and specificity is somewhat thrown out of the window, making the overall design of the site tighter and its footprint much lighter.

Let’s set up the project by creating two directories in the root; assets and static. The static directory will contain the compiled CSS and everything else that must be deployed, along with the HTML, for the website to work. While the assets folder contains any resources that won’t necessarily need to be deployed to the webserver (in this instance our Sass files).

There are many ways to install Sass, depending on your development environment. The easiest being if you have Gem installed on OS X. Simply open a terminal window and ask Gem to do the legwork:

gem install sass

For more installation instructions visit the Sass website.

Once Sass is installed on an environment, it’s necessary to ask it to watch for changes in the SCSS files and update the static CSS files accordingly. This is a pretty simple, one-liner in a terminal window:

sass --watch assets/sass/screen.scss:static/css/screen.css

or to watch the entire directory:

sass --watch assets/sass:static/css

Using the later, means we can include additional Sass files (like print.scss) and not have to start a separate watch process for each.

Sass should have now created a screen.css file in our /static/css/ directory. Although at this point the file is empty, we will link to it from our index.html file.

<link rel="stylesheet" type="text/css" media="all" href="static/css/screen.css" />

Base Sass files

We’re going to write our CSS in small, easy to navigate modules. This will save time in the future when making changes, and make it easier for other developers to pick up a project part way through.

Although we want to keep our modules as separate as possible, there will always be variables, functions and mixins that are used by multiple modules, so it wouldn’t make sense to include them repeatedly each time. With this in mind we create _base.scss in our /assets/sass directory. In here we keep anything that is shared over modules.

Some developers like to base their naming convention for colour variables on the modules they’re used for. Personally I like to simplify this into neutral, primary, secondary and, if need be, tertiary colour tones. It sounds like a lot, but in the long term we’ll usually find that the number of variables is vastly reduced as duplication is avoided.

So we’ll go ahead and add in our colour variables using this convention:

$color__white: #FFFFFF !default;
$color__black: #000000 !default;

$color__primary--xlight: #F70000 !default;
$color__primary--light: #DD0000 !default;
$color__primary: #C10000 !default;
$color__primary--dark: #990000 !default;
$color__primary--xdark: #330000 !default;

$color__secondary--xlight: #00F700 !default;
$color__secondary--light: #00DD00 !default;
$color__secondary: #00C100 !default;
$color__secondary--dark: #009900 !default;
$color__secondary--xdark: #003300 !default;

$color__neutral--xlight: #F7F7F7 !default;
$color__neutral--light: #DDDDDD !default;
$color__neutral: #999999 !default;
$color__neutral--dark: #666666 !default;
$color__neutral--xdark: #333333 !default;

followed by base sizing units:

$width__small: 481px !default;
$width__medium: 767px !default;
$width__large: 1200px !default;

$base__vertical-unit: 5 !default;
$base__horiz-unit: $base__vertical-unit !default;
$base__font-size: 15 !default;
$base__line: 20 !default;

and functions, like this one which calculate em sizes from pixels:

@function calc-em($target, $context: $base__font-size) {
    @return ($target / $context) * 1em;
}

We’ll also create a _config.scss file in the same directory. Later this will be used to override our default base and module variables, keeping everything we might change regularly in one place.

Finally, create a master Sass file (screen.scss). This will import all our necessary modules and will be compiled into CSS for use on our website. Start by importing the base and config modules.

@import "base";
@import "config";

Rather than asking Sass to create multiple CSS files and linking each one into our HTML document, we’re using the Sass @import directive to combine all modules into our master CSS file. The underscore prefix on the base and config modules prevents the Sass watch process creating these additional CSS files, reducing the amount of HTTP requests required and, hopefully, improving performance.

Adding modules

Create a directory in /assets/sass called modules. This will house all the additional modules we add to our site for styling the different types of content.

We’ll start by adding a CSS reset module (_reset.scss) and pasting in the reset code of our choice. In this case it’s Eric Meyer’s Reset CSS.

We’ll import it into our master CSS file in the same way as before:

@import "modules/reset";

When you save the changes to the master file, notice that your terminal process has detected changes and overwritten the screen.css file used by our webpage. Opening the file, you can see the contents of the reset ported across.

At this point we probably haven’t saved any time and definitely haven’t achieved anything extraordinary, but you can see the principle in action. Being able to import snippets of code into a master Sass file means we can easily toggle certain modules on or off as we need to.

If we create a module called _text.scss and import as before. We can then add all variables, functions and mixins specific to text styling into that module.

$text__family-sans:  "Helvetica Neue", Helvetica, Arial, sans-serif !default;
$text__family-serif: "Times New Roman", Times, Baskerville, Georgia, serif !default;

$text__size--alpha: 36 !default;
$text__size--beta: 30 !default;
$text__size--gamma: 20 !default;
$text__size--delta: 15 !default;

$text__size--alpha--desktop: 48 !default;
$text__size--beta--desktop: 36 !default;
$text__size--gamma--desktop: 24 !default;
$text__size--delta--desktop: 15 !default;

$text__line--alpha: 40 !default;
$text__line--beta: 36 !default;
$text__line--gamma: 22 !default;
$text__line--delta: 20 !default;

$text__line--alpha--desktop: 54 !default;
$text__line--beta--desktop: 40 !default;
$text__line--gamma--desktop: 30 !default;
$text__line--delta--desktop: 20 !default;

@mixin text($size, $line, $margin: 0, $family: "") {
    font-size: calc-em($size);
    margin-bottom: calc-em($margin, $size);
    line-height: ($line/$size);
    @if $family != "" {
        font-family: $family;
    }
}

html {
    font-size: $base__font-size * 1px;
}

body {
    font-size: 100%;
    line-height: $base__line/$base__font-size;
    color: $color__neutral--dark;
    font-family: $text__family-sans;
}

h1 {
    @include text($text__size--alpha, $text__line--alpha, 15);
    font-weight: bold;
    color: $color__primary;
}

h2 {
    @include text($text__size--beta, $text__line--beta, 15);
    color: $color__primary;
}

h3 {
    @include text($text__size--gamma, $text__line--gamma, 15);
    text-transform: uppercase;
}

p {
    @include text($text__size--delta, $text__line--delta, 10);
}

Here we have a mixin which allows us to include all our text styling on one line and variables, which allow us to change CSS values quickly and easily. Each variable is given a !default flag, which allows us to override them in our _config.scss file:

$text__family-sans: "Proxima Nova", "Helvetica Neue", Helvetica, Arial, sans-serif;
// $text__family-serif: "Times New Roman", Times, Baskerville, Georgia, serif !default;

// Base font-sizes
$text__size--alpha: 30;
$text__size--beta: 24;
$text__size--gamma: 15;
// $text__size--delta: 15 !default;

$text__size--alpha--desktop: 36;
$text__size--beta--desktop: 28;
$text__size--gamma--desktop: 15;
// $text__size--delta--desktop: 15 !default;

// // Base line-heights
$text__line--alpha: 32;
$text__line--beta: 26;
$text__line--gamma: 20;
// $text__line--delta: 20 !default;

$text__line--alpha--desktop: 38;
$text__line--beta--desktop: 32;
$text__line--gamma--desktop: 20;
// $text__line--delta--desktop: 20 !default;

As we do not need to change the values for some variables, we have no need to override them in our configuration file but, personally, I prefer to include them, commented out, so I can easily see that the value is being set by the module.

// $base__line: 20 !default;
// $base__font-size: 15 !default;

You might ask what the point of declaring and then overriding these variables is, and for one-off use there really isn’t any. But what we now have is the beginnings of a modular Sass framework that we can use again and again as a basis for any front-end project.

As we add more elements to our system, we add more modules and import as we wish.

Make it easy to understand

With the framework foundations built and modules being added as and when we need them, you could be forgiven for thinking that any developer could pick up your work and understand it almost instantly.

But that’s not always the case. Fortunately there are ways to make it even easier to understand. Firstly, commenting our Sass files using double forward slashes will not add any weight to our compiled CSS file, and so we can write as much as we like to be helpful.

...

/////////////////////////////////
// Mixins and Functions        //
/////////////////////////////////

// ## Includes standard text styling properties.
// $size: (required) Required font size in pixels
// $line: (required) Required line height in pixels
// $family (optional) Font family stack string
@mixin text($size, $line, $margin: 0, $family: "") {
...

Secondly, as mentioned before, we can use the BEM methodology. Using BEM we can include a Sass module for lists where inheritance is obvious.

.list {
    padding-left: calc-em(10);
}

.list__item {
    margin-bottom: calc-em(10);
}

.list__item--end {
    margin-bottom: 0;
}

Due to our reset module, the items within this list would not have any markers. So what if wanted to use simple bullets?

We could just add list-style-type: disc; to the styles for the .list class, but that would mean that all lists would display bullets as soon as they are assigned the base list class.

Instead we use a modifier and include that class on the unordered list element.

<ul class="list list--bullet">
    ...
</ul>
.list--bullet {
    list-style-type: disc;
    padding-left: calc-em(25);
}

Then if we need the list items within the bulleted class to behave differently, we can use nesting to ensure it does. 

.list--bullet {
    list-style-type: disc;
    padding-left: calc-em(25);

    .list__item {
        margin-bottom: calc-em(5);
    }
}

It’s not advisable to exceed 2 levels of nesting, or your CSS could become highly-specific, leading to a recurrence of the problems we’re trying to overcome; bloated page weight and ease of code navigation.

Putting it all together

Using the styles we have created, let’s put together a very simple page base on one of my favourite subjects.

<article class="layout__container">
    <header>
        <h1>Hair styling for beginners</h1>
        <h2>Tips and tricks on creating a masterful quiff.</h2>
    </header>
    <section>
        <p>Getting your hair to stand up, and stay up during a busy day in the office can be a bit of a
            hassle at times. But here&rsquo;s a full-proof method to achieving follicular greatness.</p>
        <hr />
        <figure class="media media--full">
            <img class="media__item" src="http://static.fashionbeans.com/wp-content/uploads/2011/03/quiffmain1.jpg" />
            <figcaption class="media__caption">Dude with a quiff</figcaption>
        </figure>
        <hr />
        <h3>Requirements</h3>
        <ul class="list list--bullet">
            <li class="list__item">Hair dryer</li>
            <li class="list__item">Styling putty</li>
            <li class="list__item">Finishing spray</li>
        </ul>
        <h3>Method</h3>
        <ol class="list list--number list--method">
            <li class="list__item">Wash and condition your hair as normal.</li>
            <li class="list__item">Towel dry you hair until it is only slightly damp.</li>
            <li class="list__item">Rub a small amount of styling putty between the tips of your fingers and style your quiff.</li>
            <li class="list__item">Blow dry into shape, using a brush to pull your hair up.</li>
            <li class="list__item">Add shine and extra hold by applying some finishing spray.</li>
        </ol>
    </section>
</article>

In our browser our HTML page now needs a bit of styling:

So we will go ahead and create an additional module for media and include additional styles for the number and method list modifiers.

_media.scss

_lists.scss

$list_number-color: #CC0000 !default;

// Number modifier
.list--number {
    counter-reset: items;

    .list__item {
        padding-top: calc-em(5);

        &:before {
            counter-increment: items 1;
            content: counter(items, decimal) ".";
            color: $list__number-color;
        }
    }
}

// Method modifier
.list--method {
    @extend .list--number;

    .list__item {
        position: relative;
        padding-left: calc-em(25);
        margin-bottom: calc-em(15);

        &:before {
            @include text($text__size--beta, $text__line--beta);
            font-style: italic;
            position: absolute;
            left: 0;
            top: 0;
        }
    }
}

At this point, the content of these files shouldn’t need any explanation as we’re using the same principles as previously with default variables within modules.

We can also override the default variables from our Sass configuration module, just to get it how we like it. These are just the one’s I tweaked, but feel free to play around.

$layout__bg-size: cover;

The next step

Obviously this is a very simple example of how we can build our own Sass and BEM based front-end framework, but the opportunity here is huge.

Even though there are already good frameworks out there, like Inuit and Gumby, which can do all this work for us, I’ve always been an advocate of writing your own code, whether that’s personally, or part of a development team. That way, we know exactly what’s going on, changes aren’t difficult to implement, and when something goes wrong, which it no doubt will, we’ll be more inclined to fix it.

By using this method we can build a starting block to use on all our projects, however large or small, which we can improve over time as and when new modules are written.

Erskine’s UP2.0 framework is exactly this, a living framework, that will never be ‘finished’. It’s current version is available on Github, but why not just build your own?
